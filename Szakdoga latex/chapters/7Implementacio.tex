\subsection{Az adatbetöltési fázis}

Az \texttt{ingest.py} a nyers CSV-k DuckDB-be töltéséért felel. A kapcsolatot 6 szálra és 10~GB memórialimitre állítom (\texttt{SET threads TO 6}, \texttt{SET memory\_limit = '10GB'}). A CSV-ket a DuckDB \texttt{read\_csv} függvénye streaming módban olvassa, explicit oszloptipusokkal (\texttt{UINTEGER}, \texttt{UTINYINT}, \texttt{USMALLINT}) a memóriahasználat minimalizálására:

\begin{lstlisting}[language=SQL, caption={A \texttt{mobility1} tábla létrehozása streaming CSV-olvasással.}, label={lst:mobility_load}, basicstyle=\small\ttfamily, breaklines=true, frame=single]
CREATE TABLE mobility1 AS
SELECT * FROM read_csv(
    'yjmob100k-dataset1.csv',
    columns = {
        'uid': 'UINTEGER',  'd': 'UTINYINT',
        't':   'UTINYINT',  'x': 'USMALLINT',
        'y':   'USMALLINT'
    },
    header = true
)
\end{lstlisting}

\subsection{Mobilitási metrikák számítása}

A \texttt{features.py} kétfázisú köztes-tábla stratégiát alkalmaz: először két kis staging tábla épül (\texttt{\_cell\_counts}, \texttt{\_slot\_counts}), amelyek a 111~M soros forrást kb.\ 16-szorosára tömörítik. Minden metrika külön \texttt{CREATE TABLE AS}-szel számítódik, a záró \texttt{SELECT} ezeket kapcsolja össze -- egyetlen nagy CTE-lánc helyett, ami nagy adathalmazon memória-túllépést okozna.

Az összes kiszámított metrikát az \autoref{tab:features_metrics} foglalja össze.

A gyrációs sugár ping-súlyozott változatban, anizotrop cellaméretekkel (easting $\approx 454{,}5$~m, northing $\approx 555{,}6$~m) számítódik:

\begin{lstlisting}[language=SQL, caption={A gyrációs sugár SQL-implementációja: ping-súlyozott, anizotrop cellaméretekkel.}, label={lst:rog}, basicstyle=\small\ttfamily, breaklines=true, frame=single]
SQRT(SUM(cnt * (
    POWER((x::DOUBLE - cx) * 555.556, 2) +
    POWER((y::DOUBLE - cy) * 454.545, 2)
)) / SUM(cnt)) AS radius_of_gyration
\end{lstlisting}

A \texttt{return\_rate} egyetlen \texttt{LAG()} ablakfüggvénnyel fut a staging táblán, elkerülve a memóriaigényes önillesztést. A home- és work-cella detekció \texttt{ROW\_NUMBER()} ablakfüggvénnyel; holtverseny esetén a kisebb $(x, y)$ koordinátájú cella nyer.
\subsection{A csillagséma kialakítása}

A \texttt{schema.py} a 6.\ fejezetben tervezett három dimenzió-táblát materializálja: \textbf{dim\_time} (3\,600~sor) a 75~nap × 48~időrés minden kombinációjához tartalmaz metaadatot (óra, napszak, \texttt{dow}, \texttt{is\_weekday}, \texttt{period\_label}); $d=0$ vasárnap. \textbf{dim\_cell} (20\,146~sor) az aktív rácscellákat tartalmazza domináns POI-kategóriával és 9-zónás összevont besorolással. \textbf{dim\_user} (121\,607~sor) minden \texttt{(uid, dataset)} kombinációhoz egy sort ír le; $\approx 3\,400$ felhasználóhoz nincs stabil otthon-detekció.

\subsection{Anomáliadetekció}

Az \texttt{anomaly.py} Isolation Forest algoritmust \cite{liu2008isolation} alkalmaz: a tanítás a dataset1 normál viselkedésén történik, a pontszámozás a dataset2 vészhelyzeti időszakán ($d \geq 60$) fut. A feature-mátrix öt napi metrikából áll (\texttt{total\_pings}, \texttt{location\_diversity}, \texttt{shannon\_entropy}, \texttt{radius\_of\_gyration}, \texttt{home\_work\_dist\_m}), amelyeket StandardScaler normalizál. Paraméterek: \texttt{n\_estimators = 200}, \texttt{contamination = 0.05}, \texttt{random\_state = 42}. --- A kimenet (uid, d) szintű \texttt{anomaly\_score} és \texttt{is\_anomaly} oszlopokkal az \texttt{anomaly\_scores.parquet} fájlba kerül.
